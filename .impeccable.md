## Design Context

### Users

Jarvis Cockpit is for a technical owner/operator running coding agents across a live local fleet. The primary job is to start, understand, steer, and review autonomous work without dropping into raw transport logs or reconstructing distributed state by hand. The interface is used during active development and dogfooding, where speed, truthful status, and inspectable evidence matter more than decorative polish.

### Brand Personality

Precise, calm, capable. The product should feel like a well-made engineering instrument: direct enough for an expert, legible enough to expose unfamiliar orchestration, and trustworthy under latency or failure. Copy should be concise and operational, never chatty or promotional.

### Aesthetic Direction

Refined utilitarianism using the existing T3 Code system: DM Sans for interface text, JetBrains Mono only for identifiers and code, restrained neutral surfaces, existing semantic color tokens, and both light and dark themes. Prefer the established code-agent conversation UI over separate Jarvis-specific visual treatments. Avoid raw event dumps, dashboard-card clutter, generic AI gradients/glow, excessive badges, and decorative technical noise.

### Design Principles

1. Show intent and outcome before transport detail; raw events belong behind explicit disclosure.
2. Make distributed state truthful and glanceable: requested, running, waiting, joining, completed, or failed must never be ambiguous.
3. Treat orchestration as a first-class hierarchy with named, navigable parent and child conversations.
4. Reuse the shared code-agent chat language so improvements apply consistently to Codex, Claude, and Jarvis-backed orchestration.
5. Preserve expert evidence on demand—tool inputs, outputs, commands, anchors, and errors—without flooding the default reading path.
